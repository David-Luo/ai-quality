# 测试领域 8 维正交建模框架

> 基于 ISTQB v4.0、ISO 29119-1:2022、ISO 25010:2023 等国际标准梳理

---

## 测试的本质定义

**软件测试的本质是「质量信息获取活动」**。

```
通过各种手段（执行/分析/评审/探索）
  → 获取关于软件系统的质量信息
  → 支持质量决策（发布/修复/优化）
```

测试不产出软件功能，测试产出的是**关于软件质量的信息**。

| 测试产出物 | 本质是什么 |
|-----------|----------|
| 测试用例 | 验证意图的信息载体 |
| 测试报告 | 质量状态的信息汇总 |
| 缺陷记录 | 质量问题的信息凭证 |
| 覆盖率数据 | 验证范围的信息度量 |
| 性能基线 | 系统能力的信息参照 |

---

## 核心原理

测试活动可以在 **8 个彼此独立的维度**上取值，维度之间可自由组合，互不约束。任何一个具体测试活动都是这 8 个维度的一个**坐标点**。

**正交性验证示例**：

```
"通过 API 接口进行自动化的性能回归测试，在系统层级使用黑盒技术，在 CI 阶段执行"

维度1 质量特性：    性能效率（Performance Efficiency）
维度2 接口层：      API
维度3 测试意图：    回归（Regression）
维度4 执行方式：    自动化（Automated）
维度5 测试层级：    系统测试（System）
维度6 设计技术：    黑盒（边界值 + 等价类）
维度7 生命周期时机：CI 阆段
维度8 活动生命周期：执行阶段
```

## 维度 1：质量特性（Quality Characteristic）— 验什么质量？

基于 ISO 25010:2023 的 **9 大质量特性**：

| 质量特性 | 英文 | 典型测试形态 |
|---------|------|-----------|
| 功能适合性 | Functional Suitability | 功能测试、逻辑验证 |
| 性能效率 | Performance Efficiency | 负载/压力/容量测试 |
| 兼容性 | Compatibility | 集成测试、契约测试 |
| 交互能力 | Interaction Capability | UI 可用性、无障碍测试（原 Usability 重命名） |
| 可靠性 | Reliability | 稳定性、容错性测试 |
| 安全性 | Security | SAST/DAST/IAST/SCA |
| 可维护性 | Maintainability | 代码质量分析 |
| 灵活性 | Flexibility | 跨浏览器/跨平台测试（原 Portability 重命名） |
| 安全保障 | Safety | 故障安全测试（2023 新增） |

## 维度 2：被测接口层（Test Interface / Object Layer）— 通过什么接口施测？

| 接口层 | 技术特征 | 并发模型 |
|-------|---------|---------|
| API（HTTP/gRPC/GraphQL/TCP/MQTT/WS/SQL） | 数据契约级，无 UI | 无状态，高并发 |
| UI（Web/Mobile） | 渲染级，有 UI | 有状态，低并发 |
| 代码（Source/Bytecode） | 白盒级 | 进程内 |
| 协议/网络 | 传输层 | 依协议 |

---

## 维度 3：测试意图/目的（Testing Objective / Purpose）— 为什么测？

| 测试意图 | 含义 | ISTQB 对应 |
|---------|------|---------|
| 确认测试 | 验证缺陷已修复 | Confirmation Testing |
| 回归测试 | 验证变更未引入副作用 | Regression Testing |
| 冒烟测试 | 快速验证核心路径 | Smoke Testing |
| 验收测试 | 验证满足业务需求 | Acceptance Testing |
| 探索性测试 | 发现未知缺陷 | Exploratory Testing |
| 理性检查 | 快速局部合理性验证 | Sanity Testing |

---

## 维度 4：执行方式（Execution Mode）— 怎么执行？

| 执行方式 | 特征 |
|---------|------|
| 手动（Manual） | 人工操作，适合探索性/可用性测试 |
| 自动化（Automated） | 脚本驱动，可重复执行 |
| AI 辅助（AI-Assisted） | AI 生成/自愈/智能执行 |
| 半自动（Semi-Automated） | 人机协作（如录制回放） |

---

## 维度 5：测试层级（Test Level）— 在哪个架构层级测？

基于 ISTQB v4.0 定义的 **5 级**：

| 测试层级 | 被测对象范围 |
|---------|-----------|
| 组件测试（Component/Unit） | 单个组件/函数 |
| 组件集成测试（Component Integration） | 组件间接口 |
| 系统集成测试（System Integration） | 系统间接口 |
| 系统测试（System） | 完整系统 |
| 验收测试（Acceptance） | 用户/业务视角 |

---

## 维度 6：测试设计技术（Test Design Technique）— 怎么设计用例？

基于 ISTQB v4.0 三大类：

| 技术类型 | 典型技术 |
|---------|---------|
| 黑盒（Black-box） | 等价类划分、边界值分析、决策表、状态转换 |
| 白盒（White-box） | 语句覆盖、分支覆盖、路径覆盖 |
| 经验型（Experience-based） | 探索性测试、错误猜测、清单测试 |

---

## 维度 7：生命周期时机（SDLC Timing）— 什么时候测？

| 时机 | 对应理念 |
|------|---------|
| 需求/设计阶段 | Shift-Left（静态测试/评审） |
| 开发阶段 | 单元测试、TDD/BDD |
| 集成/构建阶段 | CI 持续测试 |
| 发布前 | 回归/验收/性能/安全 |
| 生产环境 | Shift-Right（监控/灰度/混沌工程） |

---

## 维度 8：活动生命周期状态（Activity Lifecycle State）— 活动处于什么阶段？

| 生命周期状态 | 核心关注点 | 核心产出物 | 主导领域语言 |
|-------------|-----------|-----------|-------------|
| 规划阶段 | **质量特性（需求）** | TestingObjective / QualityAttributeScope / TestPlan | TestingObjective / TestPlan / RiskPriority / QualityAttributeDefinition / RequirementTraceability |
| 设计阶段 | **测试案例** | FunctionalCase / APITestCase / UITestCase / TestScenario | FunctionalCase / ApiTestCase / UITestCase / ScenarioStep / Assertion / Locator / TestDesign |
| 执行阶段 | **测试执行** | TestExecution / ExecutionResult / ExecutionReport | Execution / ExecutionSession / ExecutionQueue / ExecutionWorker / Schedule / CITrigger |
| 完成阶段 | **测试结论** | QualityConclusion / QualityGateResult / DefectSummary | QualityGate / QualityConclusion / Defect / TrendAnalysis / CoverageMetric / RootCauseAnalysis |

### 各阶段的核心职责

```
规划阶段 → 设计阶段 → 执行阶段 → 完成阶段
    │           │           │           │
    │           │           │           │
质量特性    测试案例    测试执行    测试结论
(需求定义)  (验证规格)  (信息获取)  (决策支持)
```

#### 规划阶段：质量特性（需求）为核心

- **核心问题**：要验证什么质量特性？为什么要验证？
- **核心产出**：质量特性覆盖范围定义、测试目标、测试计划
- **关键决策**：哪些质量特性需要验证、验证优先级、资源分配
- **领域不变量**：质量特性覆盖完整性、需求追溯一致性

#### 设计阶段：测试案例为核心

- **核心问题**：如何验证这些质量特性？
- **核心产出**：测试用例规格（FunctionalCase/APITestCase/UITestCase）
- **关键决策**：用例设计技术选择、断言定义、数据准备策略
- **领域不变量**：用例覆盖质量特性、断言正确性、可执行性

#### 执行阶段：测试执行为核心

- **核心问题**：实际运行测试，获取质量信息
- **核心产出**：TestExecution 实例、ExecutionResult、执行日志、证据快照
- **关键决策**：执行策略、并行度、失败处理、重试策略
- **领域不变量**：执行完整性、结果可重复性、证据完整性

#### 完成阶段：测试结论为核心

- **核心问题**：基于执行结果得出什么结论？支持什么决策？
- **核心产出**：QualityConclusion、QualityGate结果、缺陷汇总、趋势分析
- **关键决策**：发布决策、修复优先级、质量改进方向
- **领域不变量**：结论与证据一致性、门禁阈值合规性、决策可追溯性

### 与聚合根生命周期的区别

| 对比维度 | D8 活动生命周期 | 聚合根生命周期 |
|---------|---------------|---------------|
| 描述对象 | 测试活动（动作）的进程阶段 | 领域实体（对象）的状态流转 |
| 规划阶段产出 | TestingObjective（测试目标） | 无（规划阶段不产生实体） |
| 设计阶段产出 | TestCase（测试案例） | TestCase.CREATED → TestCase.READY |
| 执行阶段产出 | TestExecution（执行实例） | TestExecution.RUNNING |
| 完成阶段产出 | QualityConclusion（测试结论） | TestExecution.COMPLETED → QualityGate.EVALUATED |

**说明**：D8 描述的是「活动进程的阶段」，每个阶段有明确的核心关注点和产出物。聚合根生命周期是实体的内部状态机，不应作为 BC 边界切分依据。

## 维度正交性证明

同一个测试活动可以在所有 8 个维度上独立取值，且组合无逻辑冲突：

| 维度 | 取值示例 A | 取值示例 B | 取值示例 C |
|------|----------|----------|----------|
| 质量特性 | 功能适合性 | 性能效率 | 安全性 |
| 接口层 | API | UI | 代码 |
| 测试意图 | 回归 | 冒烟 | 确认 |
| 执行方式 | 自动化 | 手动 | AI 辅助 |
| 测试层级 | 系统测试 | 组件测试 | 系统集成 |
| 设计技术 | 黑盒 | 经验型 | 白盒 |
| 生命周期时机 | CI 阶段 | 发布前 | 开发阶段 |
| 活动生命周期 | 执行阶段 | 设计阶段 | 完成阶段 |

每列代表一个完全合法的测试活动，维度之间无耦合依赖。

## 对战略建模（BC 划分）的启示

### 维度与 BC 边界的判定标准

一个维度要成为 BC 边界，必须满足 DDD 核心条件：

| 判定条件 | 含义 |
|---------|------|
| 统一语言分裂 | 沿该维度的不同取值，产生了完全不同的领域术语体系 |
| 领域不变量独立 | 不同取值有各自独立的业务规则和约束 |
| 变更驱动力独立 | 不同取值因不同的外部原因而独立演化 |
| 团队自治边界 | 不同取值可以由独立团队负责 |

### 各维度判定结果

| 维度 | 是否作为 BC 边界 | 在 BC 中的角色 |
|------|----------------|---------------|
| **D2 被测接口层** | ✅ 是 | 主切分维度：API BC / UI BC |
| **D1 质量特性** | ⚠️ 部分 | 仅性能/安全独立 BC；功能适合性嵌入接口层 BC |
| D4 执行方式 | ❌ 否 | BC 内部横切能力 |
| D3 测试意图 | ❌ 否 | 编排 BC 的配置属性 |
| D5 测试层级 | ❌ 否 | 平台范围 + 编排配置 |
| D6 设计技术 | ❌ 否 | BC 内部方法论属性 |
| D7 生命周期时机 | ❌ 否 | 编排 BC 的触发条件 |
| D8 活动生命周期 | ❌ 否 | 聚合根生命周期不应作为 BC 边界；四阶段核心产出物为各 BC 内部聚合根 |

**核心结论**：只有 2 个维度适合作为 BC 边界——**被测接口层**（主维度）和**质量特性**（部分，仅限性能/安全）。其余维度都应作为 BC 内部的属性或配置。

### 反模式：多维度混合切分

当前 V5.0 文档的 BC 边界混合了多个维度：

| BC | 实际使用的维度 | 问题 |
|---|---|---|
| BC1 测试设计 | 隐式 = **手动** + **功能** + **设计阶段** | 执行方式 + 质量特性 + 时机混合 |
| BC2 API 测试 | = **API 接口层** + **自动化** + **功能** | 接口层 + 执行方式 + 质量特性混合 |
| BC3 UI 测试 | = **UI 接口层** + **自动化** + **功能** | 同上 |
| BC5 性能工程 | = **性能质量特性** | 纯质量特性维度 |
| BC6 安全测试 | = **安全质量特性** | 纯质量特性维度 |
| BC4 编排执行 | = **测试意图** + **执行调度** | 测试意图 + 平台能力混合 |

### FunctionalCase 与 ApiTestCase/UITestCase 的正确关系

**API 和 UI 是功能测试的两种交互方式**，不是不同的测试类型：

```
功能测试（Functional Testing）
  │
  ├── 通过 API 交互验证 → ApiTestCase
  ├── 通过 UI 交互验证  → UITestCase
  └── 通过手动操作验证  → 手动执行
```

FunctionalCase、ApiTestCase、UITestCase 是**平级的、等价的测试资产**，而非父子层次关系。BC 边界不应沿着「交互方式」强行拆分一个领域。

### 正确原则

1. BC 边界应沿**单一维度**或**明确定义的维度组合规则**切分
2. 不同维度的取值应作为 BC 内部的**配置项或属性**，而非 BC 边界本身
3. 维度之间的正交性意味着：API 可以手动调试，功能用例也可以自动化执行，性能测试可以通过 API 也可以通过 UI
4. D8（活动生命周期）揭示了另一种可能的 BC 切分方向：按「信息获取的不同关注点」切分
