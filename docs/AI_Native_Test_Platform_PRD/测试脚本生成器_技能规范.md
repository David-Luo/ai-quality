## 1. 组件定位

### 1.1 核心职责

测试脚本生成器是 Qoder CLI 工具层（Layer 3）的核心组件之一，负责将结构化的 Markdown 测试用例转化为可执行的 Python 测试脚本。它是连接测试设计（人类可读的 Markdown 用例）与测试自动化（机器可执行的 Python 脚本）的关键桥梁。

**核心能力**：
- 解析测试用例 Markdown 文件（YAML Frontmatter + 步骤表 + 测试数据）
- 根据用例类型自动选择目标框架（Playwright / Pytest+httpx）
- 通过 AI（LLM）生成符合规范的 Python 测试脚本
- 集成知识库上下文增强生成质量
- 生成后自动进行语法验证与反模式检查
- 回写脚本路径到源用例 Markdown 的 Frontmatter

**不在本组件范围**：
- 测试用例的生成（属于 `cli-case`，M1 §2.4）
- 测试脚本的执行调度（属于 `cli-exec`，M5）
- Self-Healing 定位器注入（V1.5，US-TC-003d）
- 性能测试脚本生成（属于自定义 Agent 扩展）
- 视觉回归测试脚本生成（V2.0）

### 1.2 与相关模块的关系

## 3. F1：CLI 命令接口

### 3.1 命令语法

```bash
qoder test script [OPTIONS]

Options:
  --cases PATH        输入：用例目录或单个 .md 文件 [必填]
  --output-dir PATH   脚本输出目录（默认：tests/scripts/<module>/）
```

## 4. F2：Markdown 测试用例解析

### 4.1 输入格式规范

测试用例 Markdown 文件必须符合 L1-01 §2.4 定义的格式。本组件接受以下结构：

**YAML Frontmatter**（必填字段标注 `*`）：

```yaml
---
id: TC_001           # * 用例 ID
title: 有效用户名和密码登录成功  # * 用例标题
module: 用户认证/登录   # * 所属模块
priority: P0          # * 优先级 (P0-P3)
type: functional      # * 类型 (functional|api|ui|boundary|exception|security)
created_by: qoder-cli
created_at: 2026-04-12
updated_at: 2026-04-12
script:               # 关联脚本路径（由本组件回写）
tags: [login, smoke]
status: draft         # draft | approved | deprecated
---
```

**Markdown Body**（必需 section 标注 `*`）：

```markdown
# TC_001：有效用户名和密码登录成功

## 前提条件

- 用户 `test@example.com` 已注册，状态为激活
- 服务正常运行，登录页面可访问

## 测试步骤 *

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 打开登录页面 `/login` | 页面正常显示 |
| 2 | 在用户名输入框输入 `test@example.com` | 输入框显示输入内容 |
| 3 | 在密码输入框输入 `Password123` | 密码以掩码形式显示 |
| 4 | 点击"登录"按钮 | 发起登录请求 |
| 5 | 等待页面响应 | 跳转至 `/dashboard` |

## 输入数据

| 字段 | 值 |
|------|-----|
| 用户名 | `test@example.com` |
| 密码 | `Password123` |

## 备注

此用例为冒烟测试必执行用例。
```

### 4.2 解析逻辑

使用 `python-frontmatter` 库解析（与 L1-01 §5.2 一致）：

```python
import frontmatter

# 读取用例文件
post = frontmatter.load("tests/cases/login/TC_001_valid_login.md")

# 提取 Frontmatter 元数据
case_id = post.metadata["id"]          # TC_001
case_type = post.metadata["type"]      # functional
case_title = post.metadata["title"]    # 有效用户名和密码登录成功

# 解析 Markdown Body
body = post.content
# → 提取 ## 前提条件、## 测试步骤（表格）、## 输入数据（表格）、## 备注
```

**步骤表解析规则**：
- 识别 `## 测试步骤` section 下的 Markdown 表格
- 解析表头：支持 `步骤|操作|预期结果` 或 `#|操作|请求|预期结果`（API 用例格式）
- 输出结构化列表：`[{step_number, action, expected_result}]`
- 若表格为空或格式异常，记录错误并跳过该文件

**输入数据解析规则**：
- 识别 `## 输入数据` section 下的 Markdown 表格
- 解析为键值对字典：`{field_name: value}`
- 可选 section，不存在时返回空字典

### 4.3 数据模型

```python
@dataclass
class ParsedCase:
    """解析后的测试用例结构"""
    id: str                          # TC_001
    title: str                       # 用例标题
    module: str                      # 所属模块
    priority: str                    # P0-P3
    type: str                        # functional|api|ui|boundary|exception|security
    tags: List[str]                  # 标签列表
    status: str                      # draft|approved|deprecated
    preconditions: List[str]         # 前提条件列表
    steps: List[TestStep]            # 测试步骤列表
    test_data: Dict[str, str]        # 输入数据键值对
    notes: str                       # 备注（可选）
    source_path: str                 # 源 Markdown 文件路径
    frontmatter: Dict[str, Any]      # 完整 Frontmatter（用于回写）

@dataclass
class TestStep:
    """单个测试步骤"""
    step_number: int
    action: str                      # 操作描述
    expected_result: str             # 预期结果
    request: Optional[str] = None    # API 请求（API 用例专用）

@dataclass
class GeneratedScript:
    """生成的测试脚本"""
    case_id: str                     # 关联用例 ID
    framework: str                   # playwright | pytest
    code: str                        # Python 源代码
    output_path: str                 # 输出文件路径
    validation_status: str           # passed | failed | needs_review
    validation_errors: List[str]     # 验证错误列表
```

### 4.4 验收标准

- [ ] 正确解析符合 L1-01 §2.4 规范的 Markdown 文件，成功率 >= 95%
- [ ] 缺少必填字段（id, title, type, 测试步骤）时，跳过并输出清晰的错误提示
- [ ] 支持 API 用例的四列表格格式（步骤|操作|请求|预期结果）
- [ ] 单文件解析耗时 < 100ms

## 5. F3：框架自动选择

### 5.1 判断逻辑

按以下优先级从高到低确定目标框架：

| 优先级 | 条件 | 框架 |
|--------|------|------|
| 1 | CLI 参数 `--framework` 已指定 | 使用指定值 |
| 2 | Frontmatter `type == "api"` | `pytest`（Pytest + httpx） |
| 3 | 步骤表中包含 HTTP 动词（GET/POST/PUT/DELETE/PATCH）或 `/api/` 路径 | `pytest` |
| 4 | Frontmatter `type` 为 `ui`/`functional`/`boundary`/`exception`/`security` | `playwright` |
| 5 | config.toml `[test].default_framework` 已配置 | 使用配置值 |
| 6 | 以上均不匹配 | `playwright`（默认兜底） |

### 5.2 框架对应的输出规范

| 框架 | 输出路径 | 脚本规范引用 | conftest.py 模板 |
|------|---------|-------------|-----------------|
| `playwright` | `tests/scripts/ui/test_<feature>_<case_id>.py` | L1-03 §3.1 | L1-03 §3.1.4 |
| `pytest` | `tests/scripts/api/test_<resource>_<case_id>.py` | L1-04 §3.2 | L1-04 §3.4 |

### 5.3 验收标准

- [ ] `type: api` 用例正确路由到 Pytest+httpx
- [ ] 包含 HTTP 动词的 functional 用例也正确路由到 Pytest+httpx
- [ ] `--framework` 参数可覆盖自动判断
- [ ] 判断逻辑所有分支有单元测试覆盖

## 6. F4：AI 生成 UI 测试脚本（Playwright Python）

### 6.1 脚本文件结构标准

生成的 UI 脚本必须严格遵循 L1-03 §3.1.2 定义的结构：

```python
"""
TC-ID: TC-001
Title: 用户登录 - 有效凭证
Module: auth
Generated by: qoder test script
Generated at: 2026-04-12T10:00:00Z
"""
import re
import pytest
from playwright.sync_api import Page, expect


# ─── 测试数据（内联，不使用外部文件） ───────────────────────────────────────
TEST_USER = "testuser@example.com"
TEST_PASSWORD = "Test@123456"


# ─── 测试函数 ──────────────────────────────────────────────────────────────
def test_TC_001_valid_login(page: Page, base_url: str):
    """
    测试有效凭证登录成功跳转到 Dashboard。

    前提条件：用户 testuser 已注册并激活
    """
    # Step 1: 打开登录页
    page.goto(f"{base_url}/login")
    expect(page).to_have_title(re.compile("登录"))

    # Step 2: 输入有效用户名
    page.get_by_role("textbox", name="用户名").fill(TEST_USER)

    # Step 3: 输入有效密码
    page.get_by_role("textbox", name="密码").fill(TEST_PASSWORD)

    # Step 4: 点击登录按钮
    page.get_by_role("button", name="登录").click()

    # Step 5: 验证跳转到 Dashboard
    page.wait_for_url(f"{base_url}/dashboard", timeout=5000)
    expect(page.get_by_text("欢迎")).to_be_visible()
```

### 6.2 强制规范（生成约束）

以下规范在脚本生成阶段由 Prompt 指令和 Validator 共同保障：

1. **选择器优先级**（必须从高到低使用，与 L1-03 §3.1.3 一致）：
   - `get_by_role()` — 语义化角色（推荐）
   - `get_by_label()` — 表单标签
   - `get_by_text()` — 可见文本
   - `get_by_placeholder()` — 占位符
   - `get_by_test_id()` — data-testid
   - `locator("css")` — CSS 选择器（最后手段）
   - **禁止**：`locator("xpath=...")`

2. **等待策略**：
   - 使用 `expect(locator).to_be_visible()` 等自动重试断言
   - 使用 `page.wait_for_url()` 等待导航
   - **禁止**：`page.wait_for_timeout()` 固定等待

3. **断言规范**：
   - 必须使用 `expect()` 断言（非 `assert`）
   - 每个步骤的预期结果对应一个 `expect()` 断言

4. **测试数据**：
   - 以大写常量形式内联在脚本顶部
   - 不使用外部 JSON/CSV 文件
   - 敏感数据通过 `os.environ.get("TEST_PASSWORD")` 注入

5. **命名约定**：
   - 文件名：`test_<feature>_<case_id_lower>.py`
   - 函数名：`def test_<case_id>_<slug>(page: Page, base_url: str):`

## 7. F5：AI 生成 API 测试脚本（Pytest + httpx）

### 7.1 脚本文件结构标准

生成的 API 脚本必须严格遵循 L1-04 §3.2 定义的结构：

```python
"""
TC-ID: TC-101
Title: GET /api/users/{id} - 查询存在的用户
Module: user-api
Generated by: qoder test script
Generated at: 2026-04-12T10:00:00Z
"""
import pytest
import httpx


# ─── 测试数据（内联） ──────────────────────────────────────────────────────
EXISTING_USER_ID = 1
EXPECTED_USERNAME = "testuser"


# ─── 测试函数 ──────────────────────────────────────────────────────────────
def test_TC_101_get_existing_user(api_client: httpx.Client):
    """
    GET /api/users/{id} 返回指定用户的完整信息。

    前提条件：ID 为 1 的用户已存在
    """
    # Step 1: 发送 GET 请求
    response = api_client.get(f"/api/users/{EXISTING_USER_ID}")

    # Step 2: 验证状态码
    assert response.status_code == 200, \
        f"期望 200，实际 {response.status_code}，响应体：{response.text}"

    # Step 3: 验证响应结构
    data = response.json()
    assert "id" in data, "响应缺少 id 字段"
    assert "username" in data, "响应缺少 username 字段"
    assert "email" in data, "响应缺少 email 字段"

    # Step 4: 验证数据正确性
    assert data["id"] == EXISTING_USER_ID
    assert data["username"] == EXPECTED_USERNAME
```

### 7.2 强制规范（生成约束）

与 L1-04 §3.3 一致：

1. **HTTP 客户端**：统一使用 `httpx.Client`（同步）或 `httpx.AsyncClient`（异步），**禁止使用 `requests`**

2. **断言规范**：
   - 每个断言必须有清晰的错误消息（`assert condition, "message"`）
   - 状态码断言必须是第一个断言
   - 断言顺序：状态码 → 响应结构 → 响应数据

3. **测试数据**：
   - 内联常量：适用于简单静态数据
   - `pytest.mark.parametrize`：适用于多组数据测试（如边界值）
   - 环境变量：适用于密钥、令牌等敏感数据

4. **幂等性**：
   - POST/PUT/PATCH 测试在 teardown 中通过 `try/finally` 清理创建的数据

5. **命名约定**：
   - 文件名：`test_<resource>_<case_id_lower>.py`
   - 函数名：`def test_<case_id>_<slug>(api_client: httpx.Client):`

### 8.1 验证流水线

生成的脚本在写入文件前，经过以下四层验证：

| 层级 | 检查项 | 检查工具 | 失败时处理 |
|------|--------|---------|-----------|
| L1 语法检查 | Python 语法正确性 | `ast.parse()` + `py_compile.compile()` | 自动修复重试 |
| L2 结构检查 | 恰好 1 个 `def test_*` 函数、正确的参数签名、包含 docstring、至少 1 个断言 | AST 遍历 | 自动修复重试 |

## 11. LangGraph 工作流编排

### 11.1 工作流状态

```python
from typing import TypedDict, List, Optional, Dict

class ScriptGenerationState(TypedDict):
    case_files: List[str]                    # 输入用例文件路径列表
    current_index: int                       # 当前处理的用例索引
    parsed_cases: List[ParsedCase]           # 解析后的结构化用例数据
    framework: str                           # 选定框架: "playwright" | "pytest"
    kb_context: str                          # 知识库检索结果文本
    openapi_context: Optional[str]           # OpenAPI Schema 上下文
    generated_scripts: List[GeneratedScript] # LLM 生成的脚本代码
    validation_results: List[ValidationResult]  # 语法检查结果
    retry_count: int                         # 当前重试次数
    config: Dict                             # config.toml 配置内容
    errors: List[str]                        # 累积错误消息
```

### 11.2 工作流拓扑

```python
from langgraph.graph import StateGraph, END

# 创建工作流
workflow = StateGraph(ScriptGenerationState)

# 添加节点
workflow.add_node("case_reader", case_reader_node)         # Node 1: 用例解析
workflow.add_node("framework_selector", framework_selector_node)  # Node 2: 框架选择
workflow.add_node("script_generator", script_generator_node)      # Node 3: LLM 生成
workflow.add_node("validator", validator_node)              # Node 4: 验证
workflow.add_node("case_updater", case_updater_node)        # Node 5: 写入+回写

# 添加边
workflow.set_entry_point("case_reader")
workflow.add_edge("case_reader", "framework_selector")
workflow.add_edge("framework_selector", "script_generator")
workflow.add_edge("script_generator", "validator")

# 条件路由：验证通过 → 写入 | 验证失败 → 重试或放弃
workflow.add_conditional_edges(
    "validator",
    route_after_validation,  # 路由函数
    {
        "pass": "case_updater",
        "retry": "script_generator",
        "give_up": "case_updater",
    }
)
workflow.add_edge("case_updater", END)

# 编译工作流
app = workflow.compile()
```

### 11.3 条件路由函数

```python
def route_after_validation(state: ScriptGenerationState) -> str:
    """根据验证结果决定下一步"""
    latest_result = state["validation_results"][-1]

    if latest_result.status == "passed":
        return "pass"
    elif state["retry_count"] < 3:
        return "retry"
    else:
        return "give_up"
```

### 11.4 批量处理策略

对于目录输入（多个用例文件），采用**逐个串行处理**策略：

```
对于 case_files 中的每个用例：
    1. 解析单个用例 (Node 1)
    2. 选择框架 (Node 2)
    3. LLM 生成脚本 (Node 3)
    4. 验证 (Node 4) → 可能重试回到 Node 3
    5. 写入文件 + 回写 Frontmatter (Node 5)
```

串行处理而非并行的原因：
- LLM 调用有速率限制
- 保证生成质量和一致性
- 便于用户在交互模式下逐个审核

| `playwright` | 页面结构、UI 元素角色、URL 模式、前端技术栈 | `"登录页面结构 输入框 按钮 角色"` |

每次生成执行最多 **3 个并行查询**，结果合并后截断至 **2000 tokens**，作为 LLM 上下文注入。

### 13.3 优雅降级

当知识库不可用（Qdrant 未启动、索引为空）时：
- 不中断脚本生成流程
- 输出提示信息：`ℹ 知识库不可用，跳过上下文增强`
- LLM Prompt 中省略知识库上下文部分

## 14. 错误处理与异常策略

### 14.1 错误分类与处理

| 错误类别 | 检测方式 | 响应策略 | 用户提示 |
|---------|---------|---------|---------|
| Markdown 解析失败 | `python-frontmatter` 异常或缺少必填字段 | 跳过文件，继续批量处理 | `⚠ 跳过 {filepath}: 缺少必填字段 '{field}'` |
| 步骤表解析失败 | 表格解析器异常或表格为空 | 跳过文件，继续批量处理 | `⚠ 跳过 {filepath}: 无法解析测试步骤表格` |
| LLM 调用失败 | API 超时、速率限制、网络错误 | 指数退避重试：2s, 4s, 8s（最多 3 次） | `🔄 LLM 请求失败，重试中 ({n}/3)...` |
| LLM 响应格式异常 | 无法从响应中提取 Python 代码 | 使用更严格的格式指令重新提示（1 次重试） | `⚠ LLM 返回格式异常，重试中...` |
| 脚本语法验证失败 | `ast.parse()` / `py_compile` 报错 | 自动修复循环（最多 3 次重试） | `🔧 语法错误，自动修复中 ({n}/3)...` |
| 反模式检测 | XPath / wait_for_timeout / 硬编码密码等 | 自动修复循环 | `🔧 检测到反模式: {pattern}，自动修复中...` |
| 修复重试全部失败 | 3 次重试后仍然失败 | 写入脚本并添加 `# TODO: NEEDS MANUAL REVIEW` | `❌ 无法自动修复 {case_id}，已标记待人工审核` |
| 知识库不可用 | Qdrant 连接失败或索引为空 | 优雅降级：不注入 KB 上下文，继续生成 | `ℹ 知识库不可用，跳过上下文增强` |
| 缺少运行时依赖 | playwright / httpx / pytest 未安装 | 中止并提示安装命令 | `❌ 缺少 '{pkg}'。运行: pip install {pkg}` |
| 配置文件缺失 | `tests/.qoder/config.toml` 不存在 | 使用默认配置并发出警告 | `⚠ 未找到配置。运行 'qoder config init' 创建` |
| 输出目录无写入权限 | `PermissionError` 异常 | 中止该文件，记录错误 | `❌ 无法写入 {path}: 权限不足` |

### 14.2 LLM 重试策略

```
LLM 调用
  │
  ├── 成功 → 提取 Python 代码 → 进入验证
  │
  ├── 超时 / 速率限制 → wait(2^n 秒) → 重试（最多 3 次）
  │
  ├── 网络错误 → 检查本地模型回退（Ollama）→ 重试或优雅失败
  │
  └── 响应格式异常 → 追加格式约束指令 → 重试 1 次
```

**LLM Provider 回退链**（与 L1-01 §5.3 一致）：
1. 首选：config.toml 配置的云端模型（GPT-4 / Claude）
2. 回退：本地 Ollama 模型
3. 最终：报错并提示用户检查 LLM 配置

```python
            base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",

        )
    elif provider == "azure":
        return ChatOpenAI(
            model=config.get("model"),

            base_url=config.get("base_url"),
            temperature=config.get("temperature", 0.2),
```

```python
"""
TC-ID: TC-150
Title: POST /api/users - 创建新用户成功
Module: user-api
Generated by: qoder test script
Generated at: 2026-04-20T10:00:00Z
"""
import pytest
import httpx


NEW_USER_DATA = {
    "username": "newuser_test",
    "email": "newuser_test@example.com",
    "password": "NewUser@123",
}
EXPECTED_FIELDS = {"id", "username", "email", "created_at"}


def test_TC_150_create_user_success(api_client: httpx.Client):
    """POST /api/users 创建新用户，返回 201 和用户对象。"""
    created_user_id = None
    try:
        # Step 1: 发送创建请求
        response = api_client.post("/api/users", json=NEW_USER_DATA)

        # Step 2: 验证状态码
        assert response.status_code == 201, \
            f"期望 201，实际 {response.status_code}，响应：{response.text}"

        # Step 3: 验证响应结构
        data = response.json()
        assert EXPECTED_FIELDS.issubset(data.keys()), \
            f"响应缺少字段：{EXPECTED_FIELDS - data.keys()}"

        # Step 4: 验证数据
        assert data["username"] == NEW_USER_DATA["username"]
        assert data["email"] == NEW_USER_DATA["email"]
        assert "password" not in data, "响应不应包含 password 字段"

        created_user_id = data["id"]

    finally:
        # Teardown: 删除测试用户
        if created_user_id:
            api_client.delete(f"/api/users/{created_user_id}")
```
