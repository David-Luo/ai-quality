**文档版本**：v1.0  
**所属模块**：M3 - UI 自动化测试  
**依赖模块**：M1（CLI Test Agent）、M2（测试用例管理）、M5（执行引擎）  
**最后更新**：2026-04-12

## 1. 模块定位

UI 自动化测试模块负责将测试用例转化为可执行的浏览器自动化脚本，并支持在多浏览器环境下执行、观测和分析。

**核心职责**：
- 定义 UI 自动化测试的脚本规范（Playwright Python）
- 定义 AI 生成 UI 脚本的输入输出约定
- 定义 UI 测试的执行环境配置
- 定义 UI 测试的结果收集和现场记录规范

**不在本模块范围**：
- 脚本执行调度（属于 M5 执行引擎）
- 用例存储管理（属于 M2 用例管理）
- 性能测试（不在 V1.0 范围内）

## 3. F3.1 Playwright 脚本规范

### 3.1.1 脚本文件命名约定

```
tests/scripts/ui/
├── test_<feature>_<case_id>.py   # 单用例脚本
└── conftest.py                    # 共享 fixtures（由平台生成/维护）
```

命名规则：
- 文件名：`test_` 前缀 + 功能模块名（snake_case）+ `_` + 用例 ID（小写）
- 示例：`test_login_tc_001.py`、`test_checkout_tc_012.py`

### 3.1.2 脚本文件结构标准

每个 UI 测试脚本必须遵循以下结构：

```python
"""
TC-ID: TC-001
Title: 用户登录 - 有效凭证
Module: auth
Generated by: qoder test script
Generated at: 2026-04-12T10:00:00Z
"""
import re
import pytest
from playwright.sync_api import Page, expect


# ─── 测试数据（内联，不使用外部文件） ───────────────────────────────────────
TEST_USER = "testuser@example.com"
TEST_PASSWORD = "Test@123456"
BASE_URL = "http://localhost:3000"  # 从 conftest 注入的 base_url 覆盖


# ─── 测试函数 ──────────────────────────────────────────────────────────────
def test_TC_001_valid_login(page: Page, base_url: str):
    """
    测试有效凭证登录成功跳转到 Dashboard。

    前提条件：用户 testuser 已注册并激活
    """
    # Step 1: 打开登录页
    page.goto(f"{base_url}/login")
    expect(page).to_have_title(re.compile("登录"))

    # Step 2: 输入有效用户名
    page.get_by_role("textbox", name="用户名").fill(TEST_USER)

    # Step 3: 输入有效密码
    page.get_by_role("textbox", name="密码").fill(TEST_PASSWORD)

    # Step 4: 点击登录按钮
    page.get_by_role("button", name="登录").click()

    # Step 5: 验证跳转到 Dashboard
    page.wait_for_url(f"{base_url}/dashboard", timeout=5000)
    expect(page.get_by_text("欢迎")).to_be_visible()
```

### 3.1.3 强制规范

**必须遵守**：
1. **选择器优先级**（从高到低）：
   - `get_by_role()` — 语义化角色（推荐，可访问性最佳）
   - `get_by_label()` — 表单标签
   - `get_by_text()` — 可见文本
   - `get_by_placeholder()` — 占位符
   - `get_by_test_id()` — data-testid 属性（需要前端配合）
   - `locator("css")` — CSS 选择器（最后手段）
   - **禁止**：`locator("xpath=...")` 除非必要

2. **等待策略**：
   - 使用 `page.wait_for_url()` 等待导航
   - 使用 `expect(locator).to_be_visible()` 等待元素（自动重试）
   - **禁止**：`page.wait_for_timeout()` 固定等待

3. **断言规范**：
   - 必须使用 `expect()` 断言（非 `assert`）
   - 每个步骤的预期结果对应一个 `expect()` 断言
   - 断言消息要清晰描述预期状态

4. **测试数据**：
   - 测试数据以常量形式内联在脚本顶部
   - 不使用外部 JSON/CSV 文件（简化依赖）
   - 敏感数据通过环境变量注入（`os.environ.get("TEST_PASSWORD")`）

### 3.1.4 conftest.py 标准模板

```python
"""
conftest.py - 由 qoder test script --init 生成，可手动修改
"""
import os
import pytest
from playwright.sync_api import Browser, BrowserContext, Page, Playwright, sync_playwright


def pytest_addoption(parser):
    parser.addoption("--base-url", default="http://localhost:3000")
    parser.addoption("--browser-name", default="chromium", 
                     choices=["chromium", "firefox", "webkit"])
    parser.addoption("--headed", action="store_true", default=False)
    parser.addoption("--slowmo", type=int, default=0)
    parser.addoption("--screenshot-dir", default="tests/reports/screenshots")


@pytest.fixture(scope="session")
def base_url(pytestconfig):
    return pytestconfig.getoption("--base-url")


@pytest.fixture(scope="session")
def browser_name(pytestconfig):
    return pytestconfig.getoption("--browser-name")


@pytest.fixture(scope="session")
def playwright_instance():
    with sync_playwright() as p:
        yield p


@pytest.fixture(scope="session")
def browser(playwright_instance, browser_name, pytestconfig):
    headed = pytestconfig.getoption("--headed")
    slowmo = pytestconfig.getoption("--slowmo")
    browser = getattr(playwright_instance, browser_name).launch(
        headless=not headed,
        slow_mo=slowmo,
    )
    yield browser
    browser.close()


@pytest.fixture
def context(browser: Browser):
    ctx = browser.new_context(
        viewport={"width": 1280, "height": 720},
        locale="zh-CN",
    )
    yield ctx
    ctx.close()


@pytest.fixture
def page(context: BrowserContext) -> Page:
    page = context.new_page()
    yield page
    page.close()


# ─── 失败自动截图钩子 ──────────────────────────────────────────────────────
@pytest.hookimpl(tryfirst=True, hookwrapper=True)
def pytest_runtest_makereport(item, call):
    outcome = yield
    rep = outcome.get_result()

    if rep.when == "call" and rep.failed:
        page: Page = item.funcargs.get("page")
        if page:
            screenshot_dir = item.config.getoption("--screenshot-dir")
            os.makedirs(screenshot_dir, exist_ok=True)
            screenshot_path = f"{screenshot_dir}/{item.name}_fail.png"
            try:
                page.screenshot(path=screenshot_path, full_page=True)
                # 将截图路径注入到 report 中，供执行引擎读取
                item._screenshot_path = screenshot_path
            except Exception as e:
                print(f"截图失败: {e}")
```

## 4. F3.2 AI 生成 UI 脚本

### 4.1 输入规范

CLI 命令：`qoder test script --case <case-id> [--target-url <url>]`

**输入来源（按优先级）**：

1. **测试用例 Markdown 文件**（主要输入）
   - 测试步骤（Steps）
   - 预期结果（Expected Results）
   - 输入数据（Test Data）
   - 前提条件（Preconditions）

2. **知识库上下文**（辅助输入）
   - 工程内 `.md` 文件中的页面结构描述
   - 历史成功脚本中的选择器模式
   - 被测系统的 URL 结构

3. **目标环境配置**（`tests/.qoder/config.toml`）
   ```toml
   [ui]
   base_url = "http://localhost:3000"
   default_browser = "chromium"
   headless = true
   screenshot_on_fail = true
   ```

### 4.2 AI 生成提示词结构

```
系统角色：你是一个专业的 Playwright Python 测试工程师。

上下文：
- 被测系统：{system_description}
- 技术栈：{tech_stack}
- 已知页面结构：{page_knowledge}

任务：根据以下测试用例，生成符合规范的 Playwright Python 脚本。

测试用例：
{case_content}

输出要求：
1. 严格遵循脚本结构标准（见规范文档）
2. 优先使用语义化选择器（get_by_role, get_by_label）
3. 每个测试步骤对应一个操作+一个验证
4. 测试数据以常量内联
5. 只输出 Python 代码，不添加解释
```

### 4.3 输出规范

**生成输出**：
- 文件路径：`tests/scripts/ui/test_<feature>_<case_id_lower>.py`
- 文件内容：符合 F3.1 规范的完整 Python 脚本

**生成后 CLI 交互**：
```
✓ 脚本已生成：tests/scripts/ui/test_login_tc_001.py

预览：
─────────────────────────────────────────
def test_TC_001_valid_login(page: Page, base_url: str):
    page.goto(f"{base_url}/login")
    page.get_by_role("textbox", name="用户名").fill(TEST_USER)
    ...
─────────────────────────────────────────

操作选项：
  [r] 直接运行此脚本验证
  [e] 打开编辑器手动调整
  [a] 接受并继续生成下一个
  [q] 退出

请选择 [r/e/a/q]：
```

### 4.4 验收标准

- [ ] 生成的脚本语法正确，可被 pytest 发现并执行
- [ ] 生成的脚本中不包含 XPath 选择器（除非用例要求）
- [ ] 生成的脚本中不包含固定等待（`wait_for_timeout`）
- [ ] 测试数据以常量内联，不引用外部文件
- [ ] 脚本头部注释包含 TC-ID、Title、Generated at

### 5.1 支持的浏览器

### 5.2 配置文件规范

`tests/.qoder/config.toml`：

```toml
[project]
name = "MyApp Tests"
base_url = "http://localhost:3000"

[ui]
default_browser = "chromium"    # chromium | firefox | webkit
headless = true                 # CI 环境必须为 true

viewport_width = 1280
viewport_height = 720

video_on_fail = false           # 失败录制视频（V1.5+）

[ui.retry]
max_retries = 1                 # 脚本级失败重试次数
retry_on = ["timeout", "navigation"]

[ui.network]
ignore_https_errors = false
extra_http_headers = {}
```

### 6.1 现场记录内容

| 内容 | 格式 | 触发时机 |
|------|------|---------|
| 失败截图 | PNG | 断言失败时自动 |
| 完整页面截图 | PNG（full_page=True） | 失败时自动 |
| 控制台日志 | 文本 | 全程收集 |
| 网络请求日志 | 文本 | 全程收集（可选） |
| 步骤截图 | PNG | CLI 调试模式（`--debug`） |

### 6.2 截图存储规范

```
tests/reports/<execution-id>/
├── screenshots/
│   ├── test_login_tc_001_fail.png      # 失败截图
│   └── test_checkout_tc_012_fail.png
├── console_logs/
│   ├── test_login_tc_001.txt
│   └── test_checkout_tc_012.txt
└── summary.md
```

### 6.3 控制台日志收集

在 conftest.py 中注册页面事件监听：

```python
@pytest.fixture
def page(context: BrowserContext) -> Page:
    page = context.new_page()
    
    # 收集浏览器控制台日志
    console_messages = []
    page.on("console", lambda msg: console_messages.append(
        f"[{msg.type.upper()}] {msg.text}"
    ))
    
    page._console_messages = console_messages
    yield page
    page.close()
```

执行引擎在测试结束后读取 `page._console_messages`，写入日志文件。

### 6.4 现场记录与 Bug 联动

当 `qoder test run --bug` 运行并发现失败时，执行引擎自动：
1. 读取失败截图路径（`item._screenshot_path`）
2. 读取控制台日志
3. 将截图 Base64 编码嵌入 Bug Markdown 的 `## 附件` 章节

```markdown
## 附件

### 失败截图

![失败截图](../reports/exec-001/screenshots/test_login_tc_001_fail.png)

### 控制台日志

```
[ERROR] Failed to load resource: net::ERR_CONNECTION_REFUSED
[WARN] Uncaught TypeError: Cannot read property 'name' of undefined
```
```

## 7. F3.5 脚本健康检查（V1.5）

### 7.1 功能描述

定期或在 CI/CD 流程中执行脚本健康检查，识别可能"腐化"的脚本。

### 7.2 腐化脚本识别规则

| 规则 | 描述 | 严重程度 |
|------|------|---------|
| SELECTOR_FRAGILE | 脚本使用 XPath 或绝对 CSS 路径 | WARNING |
| HARD_WAIT | 脚本包含 `wait_for_timeout` | WARNING |
| MISSING_ASSERTION | 测试步骤缺少对应的 `expect()` 断言 | ERROR |
| NEVER_EXECUTED | 脚本创建超过 30 天未执行 | INFO |
| CONSECUTIVE_FAIL | 连续失败超过 3 次（平台侧记录） | WARNING |

### 7.3 CLI 命令

```bash
qoder test health [--fix]     # 检查所有脚本健康状态
qoder test health <script>   # 检查单个脚本
```

输出示例：
```
✓ test_login_tc_001.py          OK
⚠ test_checkout_tc_012.py      WARNING: SELECTOR_FRAGILE (line 23)
✗ test_payment_tc_031.py       ERROR: MISSING_ASSERTION (3 steps)

Summary: 1 OK, 1 WARNING, 1 ERROR
建议运行 'qoder test health --fix' 自动修复 WARNING 级别问题
```

## 8. F3.6 自我修复（Healer，V1.5）

### 8.1 功能描述

当 UI 测试脚本因页面结构变化（如 DOM 变更、文案调整）导致失败时，CLI 的 Healer 节点尝试自动修复。

### 8.2 修复触发条件

- 脚本因 `TimeoutError` 或 `ElementNotFound` 失败
- 失败原因可归因于选择器失效（非逻辑错误）

### 8.3 修复工作流

```
执行失败
  ↓
分析失败原因（Analyzer 节点）
  ↓
  ├─ 选择器失效 → Healer 节点
  │     ↓
  │   截图 + 页面 HTML 快照
  │     ↓
  │   AI 重新推断选择器
  │     ↓
  │   生成修复 Patch
  │     ↓
  │   [审批] 用户确认修复
  │     ↓
  │   应用修复，标记用例 needs_review
  │
  └─ 逻辑错误 → 创建 Bug（F1.6）
```

### 8.4 修复输出

```
✗ test_checkout_tc_012.py 失败

失败原因：TimeoutError - 未找到元素 get_by_role("button", name="提交订单")

Healer 分析：
  当前页面截图显示按钮文案已变更为"确认支付"

建议修复：
  - 原：page.get_by_role("button", name="提交订单").click()
  + 修：page.get_by_role("button", name="确认支付").click()

是否应用修复？[y/N]：
```

### 11.4 视频录制（V1.5）

```python
# conftest.py（V1.5 增加）
@pytest.fixture
def context(browser: Browser, pytestconfig):
    video_dir = "tests/reports/videos"
    os.makedirs(video_dir, exist_ok=True)
    ctx = browser.new_context(
        record_video_dir=video_dir,
        record_video_size={"width": 1280, "height": 720},
    )
    yield ctx
    ctx.close()
    # 仅保留失败用例的视频
    video = ctx.pages[0].video if ctx.pages else None
    if video and not item.failed:
        os.remove(video.path())
```

```python
"""
TC-ID: TC-002
Title: 用户注册 - 邮件格式验证
Module: auth
"""
import pytest
from playwright.sync_api import Page, expect


TEST_INVALID_EMAIL = "not-an-email"
TEST_VALID_EMAIL = "newuser@example.com"
TEST_PASSWORD = "NewUser@123"


def test_TC_002_invalid_email_validation(page: Page, base_url: str):
    """注册时输入无效邮件格式，应显示验证错误。"""
    # Step 1: 打开注册页
    page.goto(f"{base_url}/register")

    # Step 2: 输入无效邮件
    page.get_by_label("邮箱").fill(TEST_INVALID_EMAIL)
    page.get_by_label("密码").fill(TEST_PASSWORD)
    page.get_by_role("button", name="注册").click()

    # Step 3: 验证错误提示
    expect(page.get_by_text("请输入有效的邮箱地址")).to_be_visible()
    expect(page).not_to_have_url(f"{base_url}/dashboard")
```

### B. 列表和分页场景

```python
"""
TC-ID: TC-015
Title: 商品列表 - 分页功能
Module: product
"""
from playwright.sync_api import Page, expect


def test_TC_015_pagination(page: Page, base_url: str):
    """商品列表分页，每页显示 10 条，翻页后内容更新。"""
    page.goto(f"{base_url}/products")

    # Step 1: 验证第一页商品数量
    product_items = page.get_by_role("listitem", name="product-item")
    expect(product_items).to_have_count(10)

    # Step 2: 记录第一页第一条商品名
    first_item_text = product_items.first.inner_text()

    # Step 3: 点击下一页
    page.get_by_role("button", name="下一页").click()
    page.wait_for_load_state("networkidle")

    # Step 4: 验证内容已更新
    new_first_item = page.get_by_role("listitem", name="product-item").first
    expect(new_first_item).not_to_have_text(first_item_text)
```

```python
"""

Module: profile

import os
```
